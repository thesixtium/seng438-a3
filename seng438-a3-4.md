EclEmma:
- Very easy to use and intuitive
- Had all coverage metrics easily accessible
- Came pre-installed

Branch:
- Helped determine if all if/else branches were covered
- Better at identifying if code is correct compared to statement
- Can be hard to achieve 100% coverage, especially as some branches are impossible to reach
- More complicated to interpret from the code coverage tool and implement

Statement:
- Easy to determine if 100% code coverage was achieved
- Doesn't indicate correctness, just if lines were ran
- Easy to interpret and implement
- Not as effective for code with complex logic

# 7 A comparison on the advantages and disadvantages of requirements-based test generation and coverage-based test generation.

Requirements-Based Testing
- Finds bugs by focusing on known functionalities
- Risk of redundant tests with little additional value
- Hard to determine exactly how many test cases are needed, especially for complicated partitions

Coverage-Based Testing
- Ensures a clear requirement for number of tests
- Provides measurable metrics to gauge testing completeness
- Tests may focus on coverage metrics over testing functionality, leading to undetected bugs

Overall, requirements-based testing and coverage-based testing address different issues when testing. It is important to use both methods to ensure complete testing of any given unit.

# 10 Comments/feedback on the lab itself

This lab was very valuable because it moved testing beyond simply writing test cases and helped us understand how to measure their effectiveness. Using coverage results to guide and enhance test design made the testing process more concrete and purposeful. It was especially helpful to see how statement, branch, and method coverage can reveal different weaknesses in a test suite, and how improving one metric does not always mean that the code is being tested well in every sense.

One strength of the lab is that it encouraged deeper engagement with the source code ( white-box). Instead of testing only from the outside, we had to analyze the internal structure of methods, identify uncovered logic, and design tests deliberately to reach missed paths, not all were able to be covered due to the way it ws intentionally designed ( not reachable). This made the exercise educational and practical, especially for understanding white-box testing and the limitations of relying only on requirements-based thinking.

A challenge, however, was that setting up and interpreting coverage tools could sometimes be confusing, especially when trying to distinguish between statement, branch, and condition-related results. In some cases, it was not immediately obvious why certain code remained uncovered or whether a missed branch was realistically reachable. More guidance or a short example walkthrough in the lab instructions could make this part easier for students and reduce time spent troubleshooting tool-related issues.

Another challenge we encountered was setting up the testing environment. Locating the required JUnit libraries that we had previously downloaded for Assignment 2 was time-consuming and somewhat distracting from the main objectives of the lab. It would be helpful if the required JUnit libraries were included directly in the lab artifacts or setup materials so that students can focus more on developing and improving their test cases rather than on environment configuration.
