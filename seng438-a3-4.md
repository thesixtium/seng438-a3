# 4 A high level description of five selected test cases you have designed using coverage information, and how they have increased code coverage

Test Case 1: RangeIntersectsDoubleTest

Target Method: intersects(double b0, double b1)

High-Level Description: This test evaluates whether a specified range [b0,b1] overlaps with the existing Range object. It specifically targets the boundaries where the input range starts before the lower bound and ends within the range, as well as where it starts within and ends after the upper bound.

Coverage Impact: This increased Branch Coverage. The original tests often only checked for full containment. By adding cases where the input range overlaps only one boundary (e.g., range(5, 10) intersecting (4, 6)), we forced the JVM to evaluate both sides of the logical OR/AND conditions within the source code.

Test Case 2: RangeIntersectsRangeTest

Target Method: intersects(Range range)

High-Level Description: Unlike the double-primitive version, this test passes a full Range object. It verifies the logic that handles null range objects and ranges that are completely disjoint (positioned entirely above or below).

Coverage Impact: This increased Line Coverage. By passing a Range object, we exercised the internal wrapper logic that extracts the lower and upper bounds from the object before passing them to the primitive intersects method. It also ensured that the "Return False" branch for non-overlapping ranges was fully executed.

Test Case 3: RangeScaleTest (Exception Handling)

Target Method: scale(Range base, double factor)

High-Level Description: This test specifically targets the error-handling logic of the static scale method. While standard tests check positive scaling, this test uses assertThrows to ensure the method correctly identifies and rejects negative scaling factors.

Coverage Impact: This increased Instruction Coverage. Most initial tests only cover the "happy path." By intentionally triggering an IllegalArgumentException, we covered the error-checking branches at the beginning of the method, which otherwise remain "red" in coverage reports.

Test Case 4: DataUtilitiesCreateArrayTest

Target Method: createNumberArray(double[] data)
High-Level Description: This test converts a 1D primitive double array into a Number (Double) object array. It verifies that the length of the resulting array is identical and that the precision of values is maintained during the boxing process.
Coverage Impact: This increased Loop Coverage. By using an array with multiple elements, we ensured that the for loop responsible for the conversion iterates more than once, covering the "continue" and "exit" conditions of the loop structure within the DataUtilities class.

Test Case 5: DataUtilitiesCreateArray2DTest

Target Method: createNumberArray2D(double[][] data)

High-Level Description: This test focuses on the conversion of nested primitive arrays. It validates the handling of rows and columns, ensuring that a 2D structure (like a matrix) is correctly mapped to a Number[][] object.

Coverage Impact: This significantly increased Complexity/Path Coverage. Converting 2D arrays involves nested loops. This test ensures that the inner loop (columns) is fully executed for every iteration of the outer loop (rows), covering edge cases where rows might be of different lengths (ragged arrays) if the implementation allows it.

EclEmma:
- Very easy to use and intuitive
- Had all coverage metrics easily accessible
- Came pre-installed

Branch:
- Helped determine if all if/else branches were covered
- Better at identifying if code is correct compared to statement
- Can be hard to achieve 100% coverage, especially as some branches are impossible to reach
- More complicated to interpret from the code coverage tool and implement

Statement:
- Easy to determine if 100% code coverage was achieved
- Doesn't indicate correctness, just if lines were ran
- Easy to interpret and implement
- Not as effective for code with complex logic

# 7 A comparison on the advantages and disadvantages of requirements-based test generation and coverage-based test generation.

Requirements-Based Testing
- Finds bugs by focusing on known functionalities
- Risk of redundant tests with little additional value
- Hard to determine exactly how many test cases are needed, especially for complicated partitions

Coverage-Based Testing
- Ensures a clear requirement for number of tests
- Provides measurable metrics to gauge testing completeness
- Tests may focus on coverage metrics over testing functionality, leading to undetected bugs

Overall, requirements-based testing and coverage-based testing address different issues when testing. It is important to use both methods to ensure complete testing of any given unit.

# 10 Comments/feedback on the lab itself

This lab was very valuable because it moved testing beyond simply writing test cases and helped us understand how to measure their effectiveness. Using coverage results to guide and enhance test design made the testing process more concrete and purposeful. It was especially helpful to see how statement, branch, and method coverage can reveal different weaknesses in a test suite, and how improving one metric does not always mean that the code is being tested well in every sense.

One strength of the lab is that it encouraged deeper engagement with the source code ( white-box). Instead of testing only from the outside, we had to analyze the internal structure of methods, identify uncovered logic, and design tests deliberately to reach missed paths, not all were able to be covered due to the way it ws intentionally designed ( not reachable). This made the exercise educational and practical, especially for understanding white-box testing and the limitations of relying only on requirements-based thinking.

A challenge, however, was that setting up and interpreting coverage tools could sometimes be confusing, especially when trying to distinguish between statement, branch, and condition-related results. In some cases, it was not immediately obvious why certain code remained uncovered or whether a missed branch was realistically reachable. More guidance or a short example walkthrough in the lab instructions could make this part easier for students and reduce time spent troubleshooting tool-related issues.

Another challenge we encountered was setting up the testing environment. Locating the required JUnit libraries that we had previously downloaded for Assignment 2 was time-consuming and somewhat distracting from the main objectives of the lab. It would be helpful if the required JUnit libraries were included directly in the lab artifacts or setup materials so that students can focus more on developing and improving their test cases rather than on environment configuration.
